# 聚合支付（twowg / lbzf）对接需求定稿

> 本文档汇总与产品/网关已确认口径，用于后续 **Go 后端 / 管理端 / Android** 实现与联调。  
> **商户号示例**：`20000097`（固定）。**密钥**不入库、不进本文档、不提交 Git。  
> **统一下单**：`https://twowg.lbzf.xyz/api/pay/create_order`  
> **查单**：`https://twowg.lbzf.xyz/api/pay/query_order`  
> 原文档：<https://twome.lubzf.com/x_mch/src/views/dev/pay_doc/pay.html>

## 1. 通道与配置

| 项 | 结论 |
|----|------|
| 商户号 | **固定** |
| 支付产品 | **不固定**为 8010。在**管理端 → 充值套餐页右上角**增加**支付配置**入口：可配置**多条**（**支付产品 ID、名称、状态、排序**）。**可同时启用多条**；用户端购买时**弹出支付方式选择**页，**选择性**使用。 |
| 支付产品 ID 规则 | **必须 4 位十进制**；**禁止**在多条配置中**重复**同一 `productId`（4）。 |
| 网关 Base URL | **长期固定**为上述地址；**切换/变更由我方**自行维护与发布。 |
| 请求形态 | 仅 **POST** + `application/x-www-form-urlencoded`，**无** JSON 体等分支。 |
| 出网/TLS 特殊要求 | **无**（5）。 |
| 超时 | **未**与对端约定，**走我方**默认（6）。 |

## 2. 签名、金额与扩展

| 项 | 结论 |
|----|------|
| 算法 | 与平台文档**严格一致**（7）。 |
| `amount` | **禁止**前导零、小数等歧义形式（8）。 |
| `extra` 空/默认 | 默认 **`-`（字符串一个横杠）**（9）。 |
| 验签与新增字段 | 对端**可能增加**字段；**验签时须支持**扩展字段**全部参与**排序签算（10）。 |
| 下单可选字段 | `returnUrl`、`clientIp`、`device`、`param1/2` 等：**不传**（11）。**联调** 对端 **不** 强制要求**补** 上述字段**（7，已确认）**。 |
| `subject` | **64 个字符**（指 Unicode 字符数）；取**后台**充值**套餐名称**；**超出截断**（2）。 |
| `body` | 与 **`subject` 同文案**（1）。 |
| `notifyUrl` | **全环境唯一、固定**（公网可访问，HTTPS 建议）；**部署时**在配置/说明中单列完整 URL 即可（3）。 |
| 下单失败判断 | **仅**依据 `retCode` 等业务字段；**与**是否返回 `payParams` **无关**（14）。 |
| `payParams` | **只**解析 **`payUrl`**；**不依赖**其他键（15）。 |

## 3. 查单

| 项 | 结论 |
|----|------|
| 主键 | **优先**使用 **`mchOrderNo`** 查询；双号**均支持**时以本项为准（16）。 |
| 查单触发 | **主要**在**用户打开订单/充值结果相关页**时**主动拉**；**不**做 App **后台定时轮询**。**须** 支持**例外**：**仅** 因**发起支付**而 **外跳**（**系统浏览器**/支付页等）**完成** 后**回到**本 App 时，**可**（**建议默认**）**拉一次** 查单（5）；**不** 因**任意** `onResume`、**非** 支付外跳的切回前台 而**自动**拉。**不**在**仅**从**首页** 等 场景**无** 支付外跳 而**自动**拉。若用户**未**回 App、仅依赖 notify，**可接受** 状态**滞后**（6）。 |
| `executeNotify` | **暂时不**做（含产品化入口）（17）。 |
| 状态映射 | 业务侧**未**单独定义「已加款」成文；**按文档** `status` **0/1/2/3** 接业务逻辑（18）。 |

## 4. 异步通知（notify）

| 项 | 结论 |
|----|------|
| 重试节奏 | 以平台为**准**，我方**不**自改（20）。 |
| 非终态多次通知 | **只更新状态**，**不**重复发权益（21）。 |
| 终态+业务已处理 | **幂等**：仍返回平台要求的成功应答；**不**再执行业务写权益（22）。 |
| 入账**唯一** 入口 | **notify** 与**查单** 两条路 **均** 进入**同一套** 幂等入账逻辑，**成功** 只**加款一次**（11）。 |
| 关单+异步**并发** | **懒关单/写取消** 与 **notify/查单 成功** **可能** 并发到达：**须** 单事务+行级约束+幂等，**避免** `cancelled` 与 `paid` 互斥失败或**重复**加币（1）。 |
| 状态**语义** | **仅** 关单、**尚未** 加币 与 **已** 入账 须**严格**区分；**未** 加币 前 **不** 将「**已关**」**误判** 为 业务**已终** 而**拒** **后一条** 真实成功（2）。 |
| notify 与查单**先后** | **以先到达成功/终态** 的处理为准；**后到的** 一路（notify 或查单）在业务**已动帐** 时**只**回平台成功应答、**不重复动帐**（7）。 |
| 关单后**晚到** 成功 | 若订单**已**因**懒检测/业务** 置为 `cancelled` 或**已** 关单，**之后** notify/查单 **仍** 返回**支付成功** 且 **验签、金额** 与**该** `mchOrderNo` 订单**一致**，**仍须** 按**成功** 入账（**幂等**，**不** 因「已关单」**拒收** **真实** 成功结果）（4）。 |
| 成功应答 | 按**平台**要求返回即可（**不**在需求层自创一套与文档冲突的返回格式；细节以**文档/联调**为准）（19+文档）。 |

## 5. 商户订单号 `mchOrderNo`

| 项 | 结论 |
|----|------|
| 总长度 | **最大 30**；**禁止 31 位及以上**；最短含最少序号时为 **20+4=24**（23）。 |
| 结构 | **前 20 位固定** + **序号段（最少 4 位，自 0001 起）**；**定宽至溢出后扩 1 位**。**序号只按「自然日 + 用户」一条线递增**（**不按** `productId` **分套**计数）：**同一用户、同一自然日** 无论换何种**支付产品**，**共用** `0001→0002→…`；前缀 20 位里**仍含** 当次下单所选的 **4 位 productId**，与序号线性**独立**（1）。 |
| 时间 8 位 | **`YYYYMMDD`（主服务器日期）**；**日界与序号「当日」**一致（**强制主服务器时区/时钟**）（24/27）。 |
| 产品 4 位 | 来自用户**在客户端选择的**、管理端已启用的**支付产品配置**（4 位 productId，**非**写死 8010）（2/24）。 |
| 用户 8 位 | 十进制 **00000001–99999999**；**超出则禁止下单**（24）。**业务**上**保证** 用户 id **不** 出现超 8 位可表示范围（3）。 |
| 并发 | 发号有**严格并发唯一**（唯一索引/原子自增/锁 等，实现时定）（28）。**任何** 时刻（含**同一毫秒内**）**不得** 产生**两笔** 相同 的 完整 `mchOrderNo`（3）。 |
| 提交网关失败 | 该**预生成**单号**作废不重用**（29），与**不可重单**、重复点击**每点一单** 一致（31）。 |
| 面额与金额 | **仅**固定面额，与上送 `amount`（分）**严格一致**；**无**非整数分、赠款/捐款（30）。**套餐价、赠送金币** 等在**用户下单创建订单时** 即**锁死**在订单行；**后台改价/改套餐/下架** **不**影响**已建** 订单的**上送**与**验签**（6）。 |
| 业务订单标识 | **对内外** 一律以 **`mchOrderNo`** 为**订单**唯一主标识；**接口路径、App、管理端、客服** 指认**同一笔** 订单时**均**使用 `mchOrderNo`；**实现上** 数据库主键**可为** 与 `mchOrderNo` **相同** 的**字符串**（**推荐**）或**保留**自增 id **仅**作内部，但**不**与 `mchOrderNo` 并列**对外**（3）。 |
| 自然 notify 重复 | 文档与 PRD 要求：**必须**防重复发权益；**不**做管理端「主动重复回调」功能（**本阶段**）（32）。 |

## 6. 管理端与对账

| 项 | 结论 |
|----|------|
| 后台导出 | **是**，需含对账/财务所需**全字段**（含双方单号、金额、状态、时间等）；**不**要求附加「原始 notify 报文列」（12）。 |
| 对账周期/流程 | **非**本技术方案，**只**保证接口/导出**可**支撑（34）。 |
| 安全 35–37 | **本阶段不**在需求文档中单独立项处理（**密钥/白名单/出网**等由运维与后续补充）。 |
| 关单与作废 | 订单创建后 **10 分钟**内未达支付成功，则**关单**；关单后该 `mchOrderNo` **不可**再**发起** 本单支付/复用（与网失败「单号不再用于新调单」**一致**)。**与 §4 配合**：**晚到** 的**真实** 成功 notify/查单 **仍** 可入账，**不** 因「**业务上** 已关单」**拒收**（见 **§4 关单后晚到成功**）。 **关单时间判断采用「懒检测」**：在用户**打开订单/充值结果/发起查单**等请求时，根据 `created_at` 与当前时间比较是否**超过 10 分钟**再置 `cancelled` 并作废单号；**不**把「全表定时扫」作**必选项**（若后续有数据治理需求再加可选任务）。**懒检测**导致长期未访问的 `pending` **长期挂库**：**可接受**（1）。**关单/取消**在**报表、对账**中记录的**时间**为**懒检测**实际**写库/更新时间**（**非**用 `created_at+10` 反推的“理论时间”）（2）。 |
| 在途**未支付** 上限 | **只卡** `date(created_at)=` **当自然日** 且**仍为** `pending` 的笔数 **< 100** 方许新建；**达 100** 则拒建。**不** 另设「**全量** 在途、含**非当日** 创建」的**总** 上限**（2）**；**他日** 挂起 之 `pending` 造成**全表** 在途 **>100**，**可接受**（4）。**已付/已关** 不占**当日在途** 槽位。 |
| 再支付/新单 | 关单/失败后再付、或需重新支付，**必须****新建**订单与**新** `mchOrderNo`；**禁止**对**同一条**业务订单**复用/续用**已作废的商户单号（4）。**已决**写死。 |
| 支付配置与 pending | 当某条**支付产品配置**被**禁用/删除**时，**依赖该** `productId` 且**仍为** `pending` 的**老订单**：**不可**再**调起** 聚合完成支付；用户**须** **新下单** 并**另选** 启用中的方式。**实现**上须 **软删+快照**；**明确禁止** 对 配置/密钥 的**硬删** 导致 订单**无法** 或 **notify/查单** **无法** 验签/匹单、**丢账**（5/9）。订单行**必** 落库 当时 `productId` 等；异步**成功** 仍**须** 能入账。（**用户侧**「不可再付」= **不** 再**发起** 新**调起**；**与** §4「关单后晚到成功」**配合**：**单号+验签+金额**。） |
| 订单标题/套餐名 | 下单时写入订单与上送 `subject`/`body` 的**套餐名称/文案** 为**快照**；**不**再随后台**改套餐名** 而**改变** 历史订单（11）。 |
| `notifyUrl` 证书与 HTTPS | 不在需求侧与台方**单独书面确认**；**以联调**能**通回调** 为准；**无**对证书品牌的**额外**强制（9） |
| Notify 回包 | 对平台的同步应答，除 **body=`success`** 外 **`Content-Type` 不**作强制约定；**以联调** 与对端**实际通过** 为准（12）。 |
| 多机时间 | **多应用机**时要求 **NTP/统一时区**（**东八**建议），与 `YYYYMMDD`+当日序号不打架。 |
| 联调/开发 | **不接受**用**同一组**生产商户密钥+回调在**开发/内网**直接联，避免污染与误刷；**内网/开发**用 **Mock 或独立配置**，与生产**隔离**（10）。 |

## 11. 与当前仓库实现差异（逻辑核对，**未改代码**）

以下为阅读 **Go `handlers/recharge_user.go` + `models/commerce.go`** 后的**现状 vs 本需求**差异，供后续改造时逐条对掉。

| 点 | 当前实现 | 本需求/目标 |
|----|----------|-------------|
| 下单与入账 | `CreateRechargeOrder` 在**同一事务**里创建订单、**直接** `wallet.Apply`、并将订单标为 `paid`（代码注释已写明：**当前无真三方、生产也即时到账**） | 真对接后应：**先** `pending`、**不**在下单事务里加币；**仅**在 **notify 验签通过 + 或查单终态** 后入账 |
| 商户单号 | `RechargeOrder` **无** `mchOrderNo` / `payOrderId` / 金额分 / 关单时间 等字段 | 需落库、与网关一致；**10 分钟关单+作废**、与发号规则配套 |
| 支付方式 | 无「支付产品配置」表与接口；`ListRechargePackages` 只返回套餐 | 需 **CRUD 支付配置** + App **选通道** + 创建订单时带 **4 位 productId** |
| 模拟支付 | 存在 `POST .../recharge-orders/:id/simulate-pay` 用于「无三方」**补入账** | **不** 将「**关闭/禁** `simulate-pay`」**与** 上线路径/发版 **强绑** 为**必做一次** 的门禁（**12**）：**由** 环境/开关/运维 **单独** 定是否生产可用；**默认** 仍 建议 生产 关闭 **防** 绕过真聚合。 |
| 金额 | 套餐 `PriceYuan` 为 `float`；上送网关需 **int 分** | 需从套餐价**固化为分**再签名/下单，**禁止**前导零等 |
| 套餐「赠送币」 | 现逻辑 `Coins+BonusCoins` 一并入钱包 | 与**支付**「固定面额」不冲突（支付侧**指人民币**金额；赠送为**产品规则**；若需禁止赠送需另开产品决策） |

**结论**：聚合支付在仓库内**尚未接入**；当前充值路径是**「伪支付」即付即到**。接网关时需**重排**创建订单、加币、**notify/查单** 与**关单** 的顺序与状态机，并**扩展**表结构与 Admin/App API。`RechargeOrder` 宜以 **`mch_order_no`（= `mchOrderNo`）** 为**主键/唯一**对外主键，与 §5 一致，**避免**与自增 `id` 双轨并行的接口形态。

## 13. 逻辑复查备忘（**继续** 梳理：非待决、偏实现/验收）

以下为通读**全文** 后仍**值得实现时留意** 的点，**不** 与上文已决口径矛盾；属**技术执行** 与**边界测试** 清单。

| 类型 | 说明 |
|------|------|
| **并发** | `懒检测关单` 与 `notify/查单` **同时** 到达时，**须** 单事务+唯一终态+幂等，**避免** `cancelled` 与 `paid` 双写或重复加币。 |
| **「先成功」**（§4） | 「**已动帐**」在**关单**场景下 往往指 **加币**；**未** 加币 前 **仅** 状态 为 关/取消 时，**首条** 成功 **notify/查单** 应 **仍** 能触发入账（**与** §4「关单后晚到」 一致，**不** 被「**先** 到的只是关单**事件**」**误判** 为 已**终态** 而 拒收 后一条）。 |
| **mch 唯一** | 序号 **日+用户** 与 **前缀含 product** 可 **同序不同前缀**，**全串** 仍 唯一；发号**须** 原子，**防** 并发生成 同 一串。 |
| **在途 100** | 仅卡 **当日创建** 之 `pending`；**他日** 挂起 的 `pending` **不** 占 当日 100，**全表** 在途 可 **>100**，属**已接受** 的 产品后果；若运营**有异议** 再 单独立规。 |
| **外跳浏览器 回拉** | Android **须** 在 **onResume/深度链接/Activity 结果** 等 **可靠** 检测「从浏览器 返回**本 App**」再**拉** 查单；**若** 用户 **未** 回到**本** App（杀进程/换机），**仍** 依赖 **订单页** 或 **notify**（§3 已接受 滞后）。 |
| **无 clientIp 等** | §2 已**不传**；**联调 已确认** 对端**不** 强制 补 上述 可选 字段（7）。**若** 对端 日后 变更 要求，**以** 联调 与 **§10 修订** 记一行 补述。 |
| **钱包/流水 引用** | 现 `WalletRef` 多 用 数字 `order_id`；**改** 主键/对外 为 `mch_order_no` 时，**外键/流水** 与 **GORM 迁移** **须** **一次性** 规划（8，见 **§11**）。 |
| **notify + 查单** | 两条 成功 路 **同** 一幂等入账，**不** 重复 加款（**见 §4 表，11**）。 |
| **无退款/无补单**（§7/§8） | **已** 与 §4「关单后晚到 仍入账」 对齐，**不** 要求 人工 补 款；**渠道** 纠纷 **见** §8 表。 |

## 14. 逐条确认清单（**一条条过会**）

> 用途：评审/上线前，由产品、后端、App、运维 **逐条** 答 **是 / 否 / 条件补充**。  
> 与上文 **有冲突** 时，以**本条会议结论**为准，并**在 §10 修订** 中记一行。  
> 当前「文档定稿方向」在括号中标注，**非** 替你代答；未开会前仍以 §1～§13 表为准。

### A. 产品与配置

1. 管理端在**充值套餐页**增加**支付配置**（多 `productId`、名称、排序、可多条**同时启用**），App 端购买前**先选支付方式** —— 是否仍确认？（定稿：**是**）
2. 多条配置中**禁止**重复同一 **4 位** 十进制 `productId` —— 是否仍确认？（定稿：**是**）
3. 套餐 **人民币售价（分）** 与**赠送币** 等在**创建订单**时**锁单**；后台**改价/改名** 不改**已建** 订单的验签与上送 —— 是否仍确认？（定稿：**是**）

### B. 商户单号与唯一性

4. `mchOrderNo` 结构为 **20 位前缀 + 序号**；序号**只** 按**自然日 + 用户** 一条线递增、**不** 按支付产品**分套**；同日换产品**共用** 序号 —— 是否仍确认？（定稿：**是**）
5. 业务上 **App 用户 id** 永不超过 8 位可表示，否则**拒建单**（不在需求内扩位） —— 是否仍确认？（定稿：**是**）
6. 发号须 **原子/唯一约束**，**任何** 时刻（**含** 同一毫秒内）**不得** 两条**相同** 完整 `mchOrderNo` —— 是否仍确认？（定稿：**是**）
7. **统一下单失败** 后，该**预落库** 单号**作废**、**不** 再**复用** 于任何新调单 —— 是否仍确认？（定稿：**是**）

### C. 在途上限与关单

8. 仅当 **`date(created_at)=` 今日** 且 仍为 `pending` 的笔数 **&lt;100** 时**才** 允许**再** 建单，达 100**拒**；**不** 对「**全历史** 在途 `pending`」做总上限 —— 是否仍确认？（定稿：**是**）
9. 因**他日** 仍 `pending` 导致**全表** 在途 **&gt;100**，**接受** 该运营/数据面后果 —— 是否仍确认？（定稿：**是**）
10. 下单后 **10 分钟** 未成功即**关单**；关单**懒检测**（用户打开**订单/结果/查单** 等**才** 写 `cancelled`），**不** 把**全表定时扫** 作**必选项** —— 是否仍确认？（定稿：**是**）
11. 报表/对账中的**关单时间** = 懒检测**实际写库**时间，**不** 用 `created_at+10` **反推** —— 是否仍确认？（定稿：**是**）

### D. 并发、终态、入账

12. `notify` 与 **查单** 在**验签+金额+单号** 对的前提下，**走同一套** 幂等入账，**只** 加款**一次** —— 是否仍确认？（定稿：**是**）
13. **关单/写取消** 与 **notify/查单 成功** 可能**并发**；实现上须 **单事务+唯一终态+幂等**，**禁止** 重复加款或**错误** 双终态 —— 是否仍确认？（定稿：**是**）
14. 订单**已** 因懒检测置为关/`cancelled`，但**尚未** 加币 时，**不** 把「**已关**」当作**已终** 而**拒** 后一条**真实** 成功；成功 **notify/查单 仍** 须能入账（与「关单后晚到成功」**一致**） —— 是否仍确认？（定稿：**是**）
15. 关单/失败后用户再付 = **新订单 + 新** `mchOrderNo`，**禁止** 复用已作废**同一** 单号**再** 调起 —— 是否仍确认？（定稿：**是**）

### E. 支付配置与数据生命周期

16. 支付配置、密钥相关记录 **只** 允许**软删+快照**；**明确禁止** **硬删** 导致 无法验签/匹单/**丢账** —— 是否仍确认？（定稿：**是**）
17. 某 `productId` 被**禁用/软删** 后，依赖它的**老** `pending` 订单**不可再调起** 完成支付，用户须**新下单** 并选**别条** 启用配置 —— 是否仍确认？（定稿：**是**；**与** 晚到成功**入账** 不矛盾：**靠** 单号+验签+金额）

### F. 网关、联调、字段

18. 统一下单 **不传** `returnUrl` / `clientIp` / `device` 等**可选** 字段；**对端不强制** 我方补传 —— 是否仍确认？（定稿：**是**；若**日后** 对端**强制** 某字段，**单开** 修订+联调，**不** 默默改表）
19. **开发/内网** 不 使用**与生产同一组** 商户号+密钥+**同** 回调在公网/内网**混联**；内网** Mock 或** 独立配置**隔离** —— 是否仍确认？（定稿：**是**）

### G. App 行为

20. 查单**主路径**：用户进**订单/结果** 相关页**主动**拉；**不** 做**后台** 定时轮询 —— 是否仍确认？（定稿：**是**）
21. **例外**拉查单：仅**因完成支付** 而**外跳**（系统浏览器/支付页等）**回到** 本 App 时**可** 拉**一次**；**不** 因**一般** 切回前台 / **任意** `onResume` **自动**拉 —— 是否仍确认？（定稿：**是**）
22. 用户**杀进程/不回到** 本 App，**仅靠** `notify` 与**日后** 再进**订单页**，**接受** 订单状态**滞后** —— 是否仍确认？（定稿：**是**）
23. 以 **WebView** 开 `payUrl` 为**主**；失败可**系统浏览器** 降级 —— 是否仍确认？（定稿：**是**）

### H. 运营、纠纷、无退款

24. **不** 提供管理端**人工改支付状态/补单** 作为正式路径 —— 是否仍确认？（定稿：**无** 该能力）
25. 渠道**重复扣款/纠纷** **无** 本系统内 工单/流程，**走** 渠道/台方/法务 —— 是否仍确认？（定稿：**是**）
26. **不** 支持**用户** 侧在 App 内**退款**；协议可一句话带过好 —— 是否仍确认？（定稿：**是**）

### I. 实现与发版

27. 若 `RechargeOrder` 对外/流水 以 **`mch_order_no`（= `mchOrderNo`）** 为**主键/唯一** 主标识，**钱包流水** 等引用 须 **一次性** 迁移、**不** 留双轨 长期并存 —— 是否仍确认？（定稿：**是** / **一次性** 规划）
28. 现有 `simulate-pay` 接口：**不** 将「**生产必关/必禁**」**与** **某次** 上线路径/发版 **强绑** 为**强制** 门禁，**而** 由**环境/开关/运维** 单独定 —— 是否仍确认？（定稿：**不** 强绑；**请再答明确**：生产环境 **default = 开 还是 关** 或 **仅 内网**？）
