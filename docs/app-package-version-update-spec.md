# 客户端版本管理 — 需求与逻辑说明（规格定稿）

> **状态**：**已实现**（2026-05-08 代码落地）；规格见下方，与代码以正文为准。  
> **全文已定稿**：含公开 API 路径与字段、双端保存文件规则、iOS 双文件校验、端别切换、兜底版本比较、种子权限与 `AllPermissions` 落点（见 §3.4～§6、§8.5、§11～§14）。

---

## 1. 目标概述

1. 管理后台菜单 **「客户端版本管理」**：维护各端发布包（后台上传至 **本服务器**）、**营销版本号**、**强制更新**、**更新说明**等；列表 **分页**、**新增**、**编辑**、**列表内切换启用/禁用**、**删除（软删）**。
2. **端别维度**：Android / iOS **各自至多一条「启用」**；启用时若同端已有启用记录则 **拒绝**，提示 **已有那一条的名称、端别、营销版本、id**（**不需要**创建时间）。
3. **软删除**：列表不可见；App 接口 **不返回**已删除记录；**允许**软删后再 **同端 + 同营销版本** 新建。
4. **App 端**：**仅冷启动** 调 **公开 GET（无登录）**。**无论** 无启用、`data` 为 null、**非法/缺失 `platform`（服务端按无更新返回，见 §5）**、还是 **请求失败**，**App 均须继续正常使用**（§7.0）。**仅当** 拿到有效启用数据且 §6.1 判定 **需更新** 时才弹窗。无更新配置时 **`Success` 且 `data` 为 `null`**，双端 **无感**。
5. **异常数据**（同端多条启用）：接口层取 **版本号最高**（§6.2）→ 相同则 **`updated_at` 最近**（打日志）。

## 2. 名词

| 名词 | 含义 |
|------|------|
| 端别 | `android` / `ios` |
| 营销版本 | Android `versionName`；iOS `CFBundleShortVersionString`；格式 **强制** `^\d+\.\d+\.\d+$` |
| 状态 | **启用** / **禁用**；新建 **默认禁用** |
| 强制更新 | §7.4 |
| 软删除 | `deleted_at` 非空 |

## 3. 管理后台 — 功能与交互

### 3.1 菜单与权限码

- 侧栏：**客户端版本管理**。
- **权限码**（`meta.perm` + 角色勾选）：**`app-release-packages`**。

### 3.2 列表（分页）与排序

- **排序**（分页前）：**启用** 在前；启用组内 **`id` 降序**；**禁用** 在后，组内 **`id` 降序**。
- **启用**：若同 `platform` 已有 **启用且未软删** 的其它记录则 **拒绝**（如 HTTP 409），提示文案需包含：**对方名称、端别、营销版本、id**。
- **禁用 / 删除**：按已定规则。

### 3.3 新增

- 公共字段：**名称**、**端别**、**营销版本**（`^\d+\.\d+\.\d+$`）、**强制更新**、**更新说明**。
- **Android**：**一个**上传控件，提交 **APK**（≤ **50MB**）。
- **iOS**：**两个**独立上传控件（**分开选文件**）：**`manifest.plist`** + **IPA**（各 ≤ **50MB**）。**保存/提交时**服务端校验 **`ipa_path` 与 `manifest_path` 均非空**。
- 新建成功即状态 **禁用**。

### 3.4 编辑

- 可改：**名称**、**端别**、**营销版本**、**强制更新**、**更新说明**；可 **单独重传** APK / IPA / plist **中的一个或多个**（不必三项全换）。
- **不可**在编辑页改 **状态**。
- **当 `platform = android` 时**：任意一次 **保存**（含仅改文案）须满足 **`apk_path` 非空**（与 iOS 双文件规则对称）。
- **当 `platform = ios` 时**：任意一次 **保存**（含仅改文案）须满足 **`ipa_path` 与 `manifest_path` 均非空**；不得出现「仅余其一」的可用记录。
- **端别切换**：保存时须满足 **目标端** 文件规则（Android：**APK**；iOS：**IPA + plist**）；缺文件则 **校验失败** 并提示先上传。**实现建议**：切换端别成功后可将 **非本端** 文件路径清空并择机删除孤立文件，避免脏数据（非产品强制，属工程卫生）。

### 3.5 存储

- 文件落 **本服务器**；单文件 **&lt; 50MB**；反向代理 **`client_max_body_size`** 等与 50MB 对齐。

---

## 4. 「各端至多一条启用」

- 仅当同端 **无其它启用未删记录** 方可启用本条；**不**自动禁用其它行。  
- 允许某端 **长期无启用** → App **永不提示**该端更新。

---

## 5. 服务端 — App 公开接口

- **路由（已定稿）**：`GET /api/v1/app/release-check?platform=android|ios`（`platform` 使用 query；实现时写入 `routes.go`，Android / iOS 客户端常量与此对齐）。
- **`platform` 缺失或非法**（非 `android` / `ios`）：**一律按「无更新」处理**，返回与无启用相同：**`utils.Success(c, nil)`**（HTTP 200，`data: null`）。**禁止**因此返回 4xx 导致客户端分支复杂化。**可选**：服务端记录 **warn 日志** 便于排错，**不向 App 用户暴露**错误提示。
- **鉴权**：无需登录。
- **无启用或无更新配置**：`utils.Success(c, nil)` → JSON 与现有封装一致：

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

- **客户端（与业务连续性，已定稿）**：
  - **`data == null`**、或 **`data` 缺少进行版本比对 / 拉包所需的字段**：**视为无更新**，**主流程照常**（不弹更新窗、不 Toast）。
  - **HTTP 非 200、超时、解析失败等业务外错误**：**静默视为无更新**，**主流程照常**（与既有「请求失败静默」一致）。
  - **仅当** 拿到 **完整有效** 的启用数据（含 `version`、本端 URL 字段、`force_update` 等）且 §6.1 判定 **需更新** 时，才进入更新弹窗；**强制更新** 亦只在此路径下出现。
  - 可不重试或仅可选一次后台重试，**不向用户 Toast**。
- **有启用**：`data` 为非空对象，**公共字段**：
  - `version`：字符串，与后台一致（`^\d+\.\d+\.\d+$`）。
  - `force_update`：**布尔** `true` / `false`（**不用** `0/1`）。
  - `release_notes`：字符串，**允许空字符串** `""`；App 弹窗若无内容可隐藏说明区或展示占位文案（实现自选）。
  - **分端 URL 字段**见 **§8.5**（仅返回与本请求 `platform` 相关字段即可，另一字段可省略，双端解析无歧义即可）。

## 6. 版本比对规则

### 6.1 App 与后台（正常比对）

- 仅 **营销版本**；忽略 build。
- 后台启用行的 `version` 必须符合 `^\d+\.\d+\.\d+$`（表单已强制）。
- 客户端：读取本地营销版本后 **trim**。若 **符合** `^\d+\.\d+\.\d+$`，与后台 `version` **字符串相等则一致**；**不等**则弹窗（含本地 &gt; 后台）。
- **本地不符合该正则**（异常包、旧规则线）：**视为与后台启用版本不一致**，**仍弹更新**（引导用户走向后台配置的包；产品选项 **A**）。

### 6.2 同端多条「启用」时的「版本最高」（服务端兜底）

- **不得**用整串字典序比较（避免 `2.9.0` &gt; `2.10.0` 错误）。
- 将 `version` 拆成三段 **非负整数** `(a,b,c)`，先比 `a`，再比 `b`，再比 `c`（与普通语义化版本一致）。
- 多条版本完全相同则再取 **`updated_at` 最近** 的一条。

## 7. App 端行为

### 7.0 与主流程关系（已定稿）

- **版本检查接口**仅为冷启动附属逻辑：**无论**「无版本 / `data` 为 null / 参数在服务端按无更新处理 / 网络或 HTTP 失败」，**App 均须继续正常使用**（进入首页与各功能，不被全屏拦截）。
- **仅**在 §6.1 判定 **需更新** 且已拿到合法 `data` 时，才弹出更新相关 UI（含强制更新全屏阻塞，见 §7.4）。

### 7.1 触发时机

- **仅冷启动**。

### 7.2 非强制更新

- **更新说明**、**立即更新**、**关闭**；关闭后 **不记忽略**，下次冷启动仍可调起。

### 7.3 立即更新

- Android：**应用内下载 APK + 安装**。
- iOS：打开服务端给出的 **企业 OTA 安装 URL**（§8.5）。

### 7.4 强制更新

- **无关闭**；**拦截返回键**（Android）及 iOS 等价返回处理。
- **全屏阻塞 App 内操作**（不可点穿到底层业务）。**不**要求、也 **不**尝试拦截系统状态栏、多任务等 **系统级 UI**（已确认 **接受**该边界）。

## 8. 安装包与分发

### 8.1 `manifest.plist`（说明）

企业 OTA 时，系统通过 **`itms-services://?action=download-manifest&url=<HTTPS 上的 plist>`** 拉取 **`manifest.plist`**（XML），再根据其内 **IPA 的 HTTPS URL** 等信息下载安装。plist 须与 IPA、Bundle ID、版本等 **一致**，且相关 URL **均为 HTTPS**。生产环境 **已具备 HTTPS**（已确认）。

### 8.2 plist 来源（已定稿）

- **由管理员在后台上传**，服务端 **不**根据 IPA **自动生成** manifest。  
- **运营责任**：plist 内填写的 IPA URL、bundle 信息等与实际发布文件 **一致**；后台 **不**自动改写 plist 内容。

### 8.3 Android

- **应用内下载 + 安装**。

### 8.5 公开 API 字段约定（已定稿）

| `platform` | 字段名 | 含义 |
|------------|--------|------|
| `android` | `download_url` | APK **HTTPS 直链**（应用内下载用） |
| `ios` | `install_url` | **完整** `itms-services://?action=download-manifest&url=...`（`url=` 后为 **经 URL 编码**的 manifest **HTTPS** 地址），客户端 **直接 `openURL`** |

- 单次请求仅针对一端，响应 **`data` 中带齐** `version`、`force_update`、`release_notes` 及 **本端对应 URL 字段**即可（不要求 Android 返回 `install_url` 或 iOS 返回 `download_url`，除非实现上希望结构化统一）。
