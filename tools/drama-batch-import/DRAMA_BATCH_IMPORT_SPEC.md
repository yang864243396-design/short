# 剧集批量导入工具方案说明（Linux / Go）

本文档固化「从 `/mnt/DuanJu/` 同步到后端 uploads、并通过管理端接口建剧建集」的完整约定，避免仅靠会话上下文遗忘。**实现状态**：后端 **YYYYMMDD** 上传路径与本仓库 Go CLI **导入闭环已对齐**；**仍未交付项**集中列于 **§18**，实施约束摘要见 **§17**。

**文档位置**：与本工具源码同在仓库目录 `tools/drama-batch-import/`。**未实现项**以本文 **§18** 为权威清单；**参考实现**与规格的逐项对照表见 **[README.md](./README.md)**「与 DRAMA_BATCH_IMPORT_SPEC 的差异说明」。**§17** 汇总方案正文中分散补写的实施细节，便于运维与代码评审一次性核对。

## 逻辑总览（端到端）

### A. 单次进程在做什么

在一台 **Linux** 机器上，进程启动后：

1. **读配置**（§2）：校验边界（§2.5）；确定是否启用 **时间过滤**、是否启用 **本轮个数上限 N**（§2.0）；stderr 打印生效策略。
2. **加锁**（§1）：保证同一时间只有一个导入实例写 `daoru.txt`、调管理端。
3. **读状态文件** `daoru.txt`（JSON Lines，§6）：得到每个剧集目录 `dir` 的历史进度。
4. **维护「已完成」的 staging**：若某 `dir` 已是 **`已完成`** 但磁盘上仍有 `<staging_root>/<dir>/`，则尝试删除（§10、§12）；删失败则下轮再试，不禁阻主流程。
5. **构造本轮候选队列**：枚举 `<source_root>` 下一级子目录 → **稳定排序**（建议字典序）→ 应用 **ModTime 过滤**（仅当传了 `mod_time_before`）→ **跳过 `daoru` 里已是 `已完成` 的 dir**（§4）。
6. **按队列逐个处理剧集目录**，直到触发 **退出条件**（见下「C」）：对每个 dir 根据 `daoru` 中 **status** 走 **拷贝 / 接口** 分支（§5、§7～§9）。
7. **退出前**：删本轮新达成 **`已完成`** 对应的 **staging** 整树（§10）；**释放锁**。

### B. 单个剧集目录内的一条流水线（抽象）

对某一 `dir`（相对 `source_root` 的子目录名）：

| 阶段 | 行为要点 |
|------|----------|
| **源 → staging** | 每次同步前 **prune**（删 staging 相对源的多余项）再 **增量拷贝**（size+mtime 一致则跳过）（§7）；**不拷贝**剧集目录下名为 **`1` / `2` / `4` / `5` 的直接子目录**（见 §7.1，因分集仅依赖 `3/*.mp4`）；对齐后进入 **`全拷贝`**（缓冲就绪）。 |
| **meta + 分类** | 读 `<staging>/<dir>/meta.json`；**seriesId** 与目录名不一致则 stderr 告警仍继续（§8）；**categories** 二次解析；分页 Ensure 分类表，缺则建（sort=10）（§8）。 |
| **封面与剧集** | 无则 **`UploadImage(cover.jpg)`** 并写入 `daoru.cover_url`；**POST 新建剧集**（字段映射 §8.4）；得到 **`drama_id`**。 |
| **分集** | 仅以 `<staging>/<dir>/3/*.mp4` 为准；小写 `.mp4`；严格 `第N集_….mp4`；重复 **N** 保留一条（**N 升序 + 文件名二级排序**）（§8.5）；逐文件 **`UploadVideo` + POST 分集**；用 **`last_ok_episode`** 续跑；免费 **N≤10**，其余 **10 金币**。 |
| **收尾** | 全部分集成功后 **`已完成`**；删 staging（退出时或启动时重试 §10）。 |

**不做 `daoru` 记录的情况**：`3/` 下无法律 mp4 → **仅 stderr**，不写 `daoru`（§5）。

### C. 本轮何时结束进程（退出条件）

以下 **先到先退**，**不得**为凑满 N 而阻塞或死循环（§2.0、§2.5）：

- 配置了 **`max_dirs_per_run = N`**：**已推进目录数达到 N** → 退出；或 **队列已无可做目录**（含时间筛后太少、`已完成` 跳过太多等）→ **立即退出**（即使 **M < N**，退出码仍宜为成功）。
- **未配置 N**：队列耗尽即退出。
- 两项均未传：**全量无时间筛**，队列耗尽退出（须 stderr 警示「无上限」）（§2.0）。

**「推进」计数**：目录被取出且至少执行了一步有效工作；**纯 `已完成` 跳过**不计入（§1）。

### D. 配置两项 optional 时的筛选 × 限额（速查）

| `max_dirs_per_run` | `mod_time_before` | 候选 | 本轮上限 |
|--------------------|-------------------|------|----------|
| 否 | 否 | 全部一级子目录 | 无 |
| 是 | 否 | 同上，不看时间 | 最多 N |
| 否 | 是 | `ModTime < 截止` | 无 |
| 是 | 是 | 同上时间条件 | 最多 N |

### E. 与后端、磁盘的关系

- **必须先**完成后端 **UploadVideo / UploadImage** 写入 **`YYYYMMDD` 子目录** 等改造并验证播放与静态访问，再上线导入工具（§9）。
- **staging** 仅作缓冲；**已完成**后删 **staging**，**绝不删** `Upload*` 落盘的正式文件（§3）。
- **401** 可自动重新登录（§1）；**multipart 上传**遇 **401** 同样清空 token、重登并重试（§17）；后续可增补「仅注册路径不上传」接口以减少孤儿文件（§10 → §18 **B1**）。

## 1. 目标与约束

- **运行环境**：Linux，与后端同机（本机路径、`127.0.0.1` 或实际监听地址调管理端 API）。
- **单次运行**：本轮实际处理目录数上限、是否按目录 `ModTime` 过滤，由 **`max_dirs_per_run`**、**`mod_time_before`** 是否传入共同决定（见 **§2.0**）；**已 `已完成` 而整段跳过的目录不计入「已取出推进」计数**；处理完本轮限额（若有）或**候选队列耗尽**即退出——**配置了 `max_dirs_per_run` 时，若可做目录数少于 N，处理完后也必须立即退出，不得等待凑满 N**（见 **§2.5**）。
- **并发**：**同一时间只允许一个导入进程**（文件锁，避免并行写 `daoru.txt` 或重复建剧）。**锁文件路径（规范）**：取 **`state_file`（`daoru.txt`）所在目录** `D`，锁文件为 **`D/.daoru_import.lock`**；与默认 `daoru.txt` 同目录时常为 `…/uploads/.daoru_import.lock`。进程正常退出须释放锁。
- **日志**：**不写独立日志文件**，仅标准输出/标准错误；「无合法分集」等情况记 stderr。
- **认证**：管理端登录 JWT；遇 **401** 允许 **自动重新登录** 后继续请求。

## 2. 启动配置（自定义）

实现可选用 **命令行参数**、**环境变量** 或 **YAML/JSON 配置文件**（优先级建议：`CLI > 配置文件 > 环境变量`，并在 `-help` / 启动横幅中写明）。

### 2.0 `max_dirs_per_run` 与 `mod_time_before` 的组合语义（均为可选）

两项 **均可不传**。解析规则：**显式传入且格式非法** → 进程退出并报错；**未传入** → 按下表语义处理（不使用该项）。

| `max_dirs_per_run` | `mod_time_before` | 候选集合（在排序、`已完成` 跳过之后） | 本轮处理上限 |
|--------------------|-------------------|--------------------------------------|--------------|
| 未传 | 未传 | **`source_root` 下全部一级子目录**（不做 `ModTime` 过滤） | **无上限**，直至本轮队列耗尽 |
| 已传 | 未传 | **同上，不做时间过滤** | **最多**推进 **N** 个目录（N=`max_dirs_per_run`） |
| 未传 | 已传 | **仅** `ModTime < mod_time_before`（解析规则见 §2.2） | **无上限**，直至符合时间的候选耗尽 |
| 已传 | 已传 | **仅**满足时间条件的目录 | **最多**推进 **N** 个目录 |

补充说明：

- **「未满 N 即退出」**：当配置了 **`max_dirs_per_run = N`** 时，经时间筛选（若启用）、`已完成` 跳过、以及本轮循环内「取出并推进」之后，若**可做目录总数不足 N**，须在处理完**当前队列中全部可做目录**后 **立即正常退出**（进程结束）；**禁止**阻塞等待、定时重扫源目录以「凑满 N」、或 busy-loop。**退出码**：若未发生未捕获故障，应以 **成功（0）** 退出——「未满 N」不是错误。
- **「遍历导入所有剧集」**（两项均未传）：单次进程内对当前候选队列持续处理直到没有可做目录； run 可能很长，启动时应 **stderr 打印「未限制 max_dirs / 未启用 ModTime 过滤」** 便于运维察觉。
- **计数**：凡计入「本轮已取出推进」的目录，仍不含「仅因 `已完成` 整段跳过」的条目（与原口径一致）。

### 2.1 `max_dirs_per_run`（本轮目录数上限，可选）

| 项 | 说明 |
|----|------|
| **含义** | 单次进程运行中，最多 **取出并至少推进一步** 的 **不同剧集目录** 个数；**未传表示不设本轮上限**（见 §2.0）。 |
| **类型** | 正整数；**边界**：须满足 **§2.5** 中的取值范围校验，否则启动失败。 |
| **示例** | `--max-dirs 10` / `IMPORT_MAX_DIRS_PER_RUN=10` |

### 2.2 `mod_time_before`（目录修改时间上限，可选）

| 项 | 说明 |
|----|------|
| **含义** | **仅当传入时**：候选一级目录的 **`ModTime` 须严格小于** 该时刻；**未传则不按修改时间过滤**（见 §2.0）。 |
| **类型** | 日历日期或日期时间字符串；**时区**为 **服务器本地时区**；解析成功后须满足 **§2.5** 日期边界（若有）。 |
| **推荐格式** | 仅日期：`2006-01-02` → 语义为 **当日 `00:00:00` 本地时间** 作为上边界，`ModTime < 该时刻`。解析失败则退出并打印示例。 |
| **示例** | `--before-date 2026-05-01` / `IMPORT_MOD_TIME_BEFORE=2026-05-01` |

### 2.3 其它常见可选启动项（建议一并支持）

实现时可与导入二进制放在同一配置结构中，便于运维：

| 配置键 | 说明 |
|--------|------|
| `source_root` | 源根路径，默认 `/mnt/DuanJu` |
| `staging_root` | staging 根路径，默认 `/www/wwwroot/short/backend/uploads/staging` |
| `state_file` | `daoru.txt` 路径，默认 `…/uploads/daoru.txt` |
| `admin_base_url` | 管理 API 根，如 `http://127.0.0.1:8080/api/v1/admin` |
| `admin_user` | 登录用户名 |
| `admin_password` | 登录口令：**明文**（环境变量 `IMPORT_ADMIN_PASSWORD` 或 CLI `-admin-password`）；勿写入世界可读配置文件 |
| `admin_password_file` | **可选**：密钥文件路径（环境变量 `IMPORT_ADMIN_PASSWORD_FILE` 或 CLI `-admin-password-file`）；读取文件全文 **`TrimSpace`** 作为口令；文件须进程可读、权限收紧。**优先级**（无配置文件层时）：**CLI 明文 > CLI 文件路径 > env 明文 > env 文件路径**。 |
| `allow_future_before_date`（实现自定命名） | **可选**：为 **true** 时 **跳过** §2.5 对 `mod_time_before`「不得明显超过当前时刻 +24h」的校验，用于业务确需远期截止日期的场景；**默认须校验**，误填未来日期应启动失败。 |

### 2.4 默认值策略

- **`max_dirs_per_run`、`mod_time_before`**：**无默认值即表示「未启用该项约束」**，语义见 **§2.0**；不得在运维不知情时静默套用隐藏的「假默认值」。
- **其它键**（如 `source_root`）：可有文档约定默认值，但启动横幅建议打印最终生效路径。
- **解析冲突**：同一键在 CLI 与配置文件同时出现时按 §2 顶部优先级覆盖；**传入但非法**（如不满足 §2.5、日期无法解析）→ **退出**。

### 2.5 边界值校验与「未满 N」退出

#### 启动阶段（参数合法性）

| 配置项 | 校验规则（建议实现） |
|--------|----------------------|
| **`max_dirs_per_run`**（已传入） | 必须为整数，且 **`1 ≤ N ≤ N_max`**。`N_max` 为编译期或配置文件常量（建议 **5000～10000**），**须在 `-help` 或启动失败信息中明示**；**超出范围 → 启动失败并 stderr 说明合法区间**。 |
| **`mod_time_before`**（已传入） | 必须按 §2.2 成功解析为时间点；建议额外校验：**解析得到的时刻 ≥ `1970-01-01 00:00:00` 本地**，且 **≤ 本地当前时刻 + 24h**（允许极小时钟误差）；**明显误填远期未来 → 启动失败**。若业务确需远期截止日期，须通过 **§2.3** 所列 **专门开关**（如 `--allow-future-before-date`）放行；**开关默认关闭**。 |

未传入的配置项 **不参与** 上述数值校验。

#### 运行阶段（候选不足仍须结束）

- 若 **`max_dirs_per_run` 已配置**：本轮从队列中取目录并推进，直到 **(a) 已推进数达到 N**，或 **(b) 队列中已无可做目录**（含：时间筛后为空、仅剩 `已完成`、或其它跳过规则导致无可处理项），**以先到者为准**；**(b) 时即使已推进数 &lt; N，也必须立即退出进程**。
- stderr 建议在退出前打印摘要：**本轮计划上限 N、实际推进数 M、退出原因（配额用尽 / 队列耗尽）**，便于审计。
- **参考日志格式（可实现选用）**：一行形如 `[drama-import] summary: planned_max=<整数或 unlimited> advanced=<整数> exit_reason=<quota_reached|queue_exhausted>`。

## 3. 路径约定

| 用途 | 路径 |
|------|------|
| 源目录（一级剧集文件夹） | `/mnt/DuanJu/<剧集目录名>/` |
| 缓冲目录（可整棵删除） | `/www/wwwroot/short/backend/uploads/staging/<剧集目录名>/` |
| 状态文件（JSON Lines） | `/www/wwwroot/short/backend/uploads/daoru.txt` |
| 上传产物（目标形态，见第 9 节） | `./uploads/videos/YYYYMMDD/…`、`./uploads/images/YYYYMMDD/…` |

缓冲目录 **仅**使用 `uploads/staging/`，与 `UploadVideo` 写入的 **`./uploads/videos/YYYYMMDD/…`** 分离；**禁止**删除带时间戳上传生成的文件。

上表路径为 **默认约定**；实际部署可通过 **§2.3** 的 `source_root`、`staging_root`、`state_file` 覆盖（缓冲目录则为 `<staging_root>/<剧集目录名>/`，状态文件路径与 `daoru.txt` 配置一致）。

**剧集目录内常见布局说明**：业务侧可能在 `<剧集目录名>/` 下并列存在 `1`、`2`、`3`、`4`、`5` 等子目录；导入链路 **仅读取 `3/` 下符合 §8.5 的 mp4** 与根部的 `meta.json`、`cover.jpg` 等。**§7.1** 约定同步到 staging 时 **省略** `1`、`2`、`4`、`5` 以节省磁盘与时间；**不影响**对 `3/` 及剧集根目录其它文件的拷贝。

## 4. 候选目录筛选与排序

- 枚举 **`source_root`**（默认 `/mnt/DuanJu/`）下**一级子目录**。
- **修改时间**：**仅当启动配置传入 `mod_time_before` 时**启用——目录 `ModTime` **严格小于** §2.2 解析得到的本地时刻；**未传 `mod_time_before` 时不做此项过滤**。
- **排序**：任意**稳定**顺序即可（建议：**目录名字典序**）。
- **`daoru.txt` 中 `已完成`**：整目录跳过，不参与拷贝与接口。

## 5. 状态机（中文状态字面量）

仅允许下列四种 `status`：

| 状态 | 含义 |
|------|------|
| `拷贝部分` | 源 → staging 尚未对齐 |
| `全拷贝` | staging 已与源一致；正在进行或尚未开始「封面 + 建剧 + 建集」（见下「状态语义补充」） |
| `接口部分报错` | 曾在接口阶段失败或中断；含「剧已建、分集未齐」等续跑场景 |
| `已完成` | 本分集链路走完；staging 应在退出或启动时删除 |

**状态语义补充（避免歧义）**：四种状态为字面量约束，但 **`全拷贝` 名称易误解**。约定如下——

- 自 staging 对齐并写入 **`全拷贝` 起**，直至 **`已完成`** 或 **`接口部分报错`** 之前，若尚未发生失败，实现可**一直维持 `全拷贝`**（即：**正在逐集 UploadVideo + CreateEpisode 的中间过程仍属于 `全拷贝`**），仅在出现异常或中断时写入 **`接口部分报错`**。
- **`接口部分报错`** 也可表示「剧已创建、分集未完」的正常续跑态，不等于 stderr 意义上的「报错」字面。

若实现上倾向「一经开始传分集即从 `全拷贝` 转入 `接口部分报错` 表示进行中」，会与「失败才写 `接口部分报错`」冲突，**不推荐**；除非扩展第五种状态（当前规格不允许）。

**无合法 `.mp4` 分集**（`staging/…/3/` 下无符合 §8.5 的文件）：

- **若 `daoru.txt` 中尚无任何关于该 `dir` 的记录**：**不新增行**，仅 stderr 记日志并跳过该目录。
- **若已存在该 `dir` 的记录**（例如历史上曾进入拷贝或接口阶段）：允许实现 **仅更新** `last_error`（及 `updated_at` 等辅助字段）以便运维辨认「反复命中且无可用分集」；**不得引入**第五种 **`status`**。该类跳过仍视为目录已被本轮 **调度尝试**，是否计入 **`max_dirs_per_run`** 见 §2.0「推进」口径（参考实现：**计数 +1**）。

**副作用**：源目录长期无合法 mp4 时，**每次运行都会再次命中该目录**直至人工移走或修正内容；若需压制重复扫描，须另做运维策略（超出本文四种状态范围）。

## 6. `daoru.txt` 格式（JSON Lines）

- **一行一条** JSON 对象。
- **`已完成` 行永久保留**，用于跳过重复处理。

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `dir` | string | 剧集目录名，与源、`staging` 子目录名一致 |
| `status` | string | 四种中文状态之一 |
| `drama_id` | number | 建剧成功后写入 |
| `cover_url` | string | `UploadImage` 成功后写入；下次复用，避免重复上传 |
| `last_ok_episode` | number | 最后一个已成功 **入库** 的分集号 **N**（来自文件名 `第N集_…`） |
| `series_id_warned` | boolean | 已对 `seriesId` ≠ 目录名做过告警则可置 true，避免刷屏 |
| `updated_at` | string | RFC3339 或等价可读时间 |
| `last_error` | string | 最近一次失败简述 |

### 进度与 DB 不一致（T2）

**始终以 `daoru.txt` + 后端行为为准**：若某 `N` 在后端已存在，`AdminCreateEpisode` 返回可辨认的业务错误（消息中含 **「已存在」** 等字样即可）——实现上对该 **N** 视为已入库成功：**将 `last_ok_episode` 更新为该 `N`** 并继续下一集；**不做启动时对账扫描 DB**。具体匹配规则由实现自定并写入运维说明。

### `daoru.txt` 读写可靠性

规格未限定实现库；建议 **写入前写临时文件再原子替换**：临时文件名为 **`{state_file}.tmp`**（与 `daoru.txt` 同目录），写入成功后 **`rename` 覆盖正式路径**，避免进程中断产生半截 JSON 行。

## 7. 拷贝流程（源 → staging）

在进入「源 → staging」同步前（含 **`拷贝部分`** 续跑，以及 **`接口部分报错`** 阶段 staging 缺文件时的补救 **T1：允许**）：

1. **Prune**：以 **裁剪后的源视图**为权威（见 **§7.1**：逻辑上等价于源目录 **去掉** 名为 `1`、`2`、`4`、`5` 的四个**直接子目录**及其整棵后代），删除 staging 中存在但该视图中不存在的路径（多余文件/目录）。  
   - 若 staging 中仍留有历史上误同步的 **`1/`、`2/`、`4/`、`5/`** 整树，亦应在本阶段 **删除**（源盘即使仍有这四个目录，staging **不要求** 与之镜像）。
2. **增量拷贝**：仅在 **裁剪后的源视图** 上逐文件拷贝；目标已存在且 **size 与 mtime** 均与源一致则 **跳过**。

对齐完成后将状态更新为 **`全拷贝`**（若尚未进入接口阶段）。

### 7.1 不拷贝子目录 `1`、`2`、`4`、`5`（与 §8.5 / `3/*.mp4` 单一数据源对齐）

**动机**：分集入库 **唯一**使用 `<剧集目录>/3/*.mp4`（§8.5）；`1`、`2`、`4`、`5` 通常为其它素材或历史目录，**不参与**上架与播放链路。

**约定**（相对 **每个** `<source_root>/<剧集目录名>/` 根目录）：

| 项 | 说明 |
|----|------|
| **作用域** | 仅针对该剧集目录下的 **直接子目录**，**基名**（最后一级目录名）**完全等于** `1`、`2`、`4`、`5` 之一（Linux 部署建议 **大小写敏感**）。 |
| **行为** | **不**将上述四个子目录及其 **任意深度后代** 纳入「源 → staging」增量拷贝；staging 侧 **不得** 因本轮同步而长期保留与之对称的 `1/`～`5/` 树（见上 **Prune**）。 |
| **仍须拷贝** | 剧集根下文件（如 `meta.json`、`cover.jpg` 等）、**整个 `3/`** 子树，以及 **不**落在上述四个排除名之下的其它相对路径（若有）。 |
| **源盘** | 源目录物理盘上 **`1`～`5` 可继续存在**；工具仅为 **staging 缓冲与上传准备** 而裁剪视图，**不删除**源盘上这四个目录及其内容。 |

**与 prune 的口径**：权威树 = 源目录 **逻辑上删除** 四个排除子目录后的树形结构；staging 中凡属于「排除子树镜像」的残留，一律按 prune 规则清理。

## 8. `meta.json` 与业务映射

### 8.1 示例结构要点

- `episode_ids`：可为数组；**分集总数以 mp4 为准**，不以该数组长度为准。
- `video.categories`：可为 **JSON 字符串再套一层数组**，例如：`"[\"都市情感\", \"虐恋\", …]"`，需 **二次解析** 得到名称列表。
- `video.title`、`video.videoDesc`、`video.playCnt`、`video.seriesId`（若有）等按后端字段使用。

### 8.2 `seriesId` 校验（T9）

若存在 `video.seriesId`：与 **目录名**比较；不一致则 **告警（stderr）仍继续**；可将 `series_id_warned` 置 true。

### 8.3 分类

- 对每个分类名：分页拉取管理端分类列表直至判定「存在或需新建」。
- **不存在则新建**：`sort` **固定为 10**；名称与解析出的字符串 **完全一致**（含空格等）。
- **新建剧集**时：`category` 字段按**当前后台规则**：**逗号拼接分类名**（不写分类 id）。

### 8.4 新建剧集（管理端）

调用现有 `POST /admin/dramas`（前缀以项目 `routes` 为准，一般为 `/api/v1/admin`）。

| 需求项 | 取值 |
|--------|------|
| 标题 | `video.title` |
| 推荐排序 | 空（`null`/省略） |
| 封面 | **`UploadImage(cover.jpg)`** 返回的 `cover_url`（持久化到 `daoru`） |
| 描述 | `video.videoDesc` |
| 分类 | 上表规则拼接名称 |
| 总集数 | **`staging/.../3/` 下将实际入库的分集数**：即严格解析文件名后 **去重后的合法 `N` 的个数**（与同目录下「重复 `N` 仅保留第一个文件」规则一致）；**不等于**目录内 `.mp4` 物理文件总数 |
| 评分 | `9.9` |
| 热度 | `video.playCnt` |
| 状态 | **`completed`**（已完结） |
| 上架 | **`enabled: false`**（下架） |

### 8.5 新建分集（管理端）

- **唯一数据源**：`staging/<剧集目录名>/3/*.mp4`。
- **扩展名**：仅 **`*.mp4`（小写）**。
- **文件名**：严格正则 **`第(\d+)集_.+\.mp4`**；不匹配 **跳过该文件**。
- **重复集号 N**：多个文件对应同一 **N** — **保留一条**（Q1 理解 A）。因列表按 **N 升序** 排列，同 **N** 的多文件需 **二级排序**（建议：**文件名字典序 UTF-8**），取排序后第一个，其余跳过。
- **排序**：先解析所有合法文件的 **N**，按 **N 数值升序** 形成列表；**「从第 37 个继续」**指该列表中的第 37 个元素，断点用 **`last_ok_episode`**（已成功入库的最大 **N**），下一步处理 **`N > last_ok_episode`**。
- **标题**：`第{N}集`。
- **免费**：**N ≤ 10** 为免费；**N > 10** 收费，`unlock_coins = 10`。
- **视频**：每个 mp4 **调用一次 `UploadVideo`**，使用返回的 `path` 写入分集的 `video_path`（或项目所需字段）；对外播放仍走现有逻辑。

## 9. 上传产物目录规则（后端改造）

- **子目录名**：上传发生时服务器**本地时区**的 **`YYYYMMDD`**（例：`20260507`）。
- **视频**：`./uploads/videos/YYYYMMDD/<毫秒或约定前缀>_原名.mp4`（与现有「时间戳+原名」策略兼容，仅多一层日期目录）。
- **封面**：`./uploads/images/YYYYMMDD/…`，返回 URL 需带 `/uploads/images/YYYYMMDD/…`。
- **视频与封面共用同一种日期目录规则**。

### 上线顺序（Q6-A）

1. **先**改后端 `UploadVideo`、`UploadImage` 及静态访问、`stream` 验签路径兼容。
2. **再**发布导入工具，对接新路径。

## 10. 接口失败与续跑

- **拷贝失败**：更新 `拷贝部分`、`last_error`，**继续下一个目录**。
- **封面已传、建剧失败**：依赖 `cover_url` 已写入 `daoru`，下次不重传封面。
- **分集中途失败**：状态 **`接口部分报错`**，更新 `last_ok_episode`、`last_error`；下次从 **`N > last_ok_episode`** 继续 **UploadVideo + CreateEpisode**。
- **`接口部分报错` 且 staging 不完整**：允许再次 **prune + 增量** 补 staging **（T1）**；与正常拷贝相同，须遵守 **§7.1**（不镜像 **`1`/`2`/`4`/`5`**）。

### 后续需求（Q6 / 孤儿文件）

- 记入产品需求：**支持「服务端文件已存在时仅注册分集、不再 multipart 上传」** 的管理端能力，以减少重试产生的孤儿上传文件；导入工具在该 API 就绪后切换优先策略。

在当前 API 未提供前：**接受**重试可能产生的多余 `uploads/videos/…` 文件，或由运维按需清理。

## 11. 进程退出与 staging 清理

- **退出前**：对本轮已标记 **`已完成`** 的目录，删除 **`/www/wwwroot/short/backend/uploads/staging/<剧集目录名>/`** 整棵树。
- **删除失败（Q7）**：**保持 `已完成`**，写 `last_error`/stderr；**下次启动**时对 `已完成` 若仍存在 staging 路径则 **重试删除**。

---

## 12. 主循环摘要

1. **解析启动配置**（§2），按 **§2.0** 解析「时间过滤是否启用」「本轮是否有目录数上限」；stderr 打印生效组合（含「未限制」明示）。
2. 获取进程锁 **（T3）**。
3. 读 `daoru.txt`；对 **`已完成`** 且 staging 仍存在 → 尝试删 staging。
4. 枚举并排序候选目录；按 §4 / §2.0 应用 ModTime 过滤（若有）；跳过 **`已完成`**。
5. 若配置了 **`max_dirs_per_run`**：本轮最多推进该数量的目录；**未配置**则直至队列耗尽。**队列提前耗尽时须立即退出**，即使未满 N（§2.5）。
6. 按状态执行：**拷贝部分** / **全拷贝** / **接口部分报错**，规则见上文。
7. 退出前删 **`已完成`** 对应 staging（失败则下轮再试）。
8. 释放锁。

## 15. 仍依赖实现核对的后端细节（非歧义，属联调清单）

- **`video.playCnt` 类型**：JSON 可能为数字或字符串；绑定时需兼容。

- **varchar(500)**：`cover_url` / `video_path` 加长日期目录后一般仍够用；超长文件名需在运维或上传侧限制。
